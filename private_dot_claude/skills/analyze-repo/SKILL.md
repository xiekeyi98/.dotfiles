# 全面代码仓库分析

对目标仓库 `$ARGUMENTS`（如未指定则为当前目录）执行全面多维度分析。

## 执行规则

1. **必须使用 10 个并行子 Agent 分别执行以下维度**，每个 Agent 独立完成自己的分析
2. **严禁提前交付** — 所有 10 个子 Agent 全部完成后，才能开始汇总报告
3. 最终报告用**中文**输出
4. 每个维度的结论必须有具体的文件路径、代码片段或数据支撑，不允许空泛描述

## 容错与跳过规则（防止死循环）

每个子 Agent **必须遵守**以下规则：

1. **前置探测**：每个 Agent 在开始分析前，先快速检测当前环境（是否为 git 仓库、是否有 commit 历史、是否有构建配置文件等）。如果缺少该维度所需的关键前提，**立即输出「该维度不适用」并说明原因**，不得反复重试或猜测。
2. **无 git 仓库 / 无 commit 历史**：
   - 如果目标目录不是 git 仓库（`git rev-parse --git-dir` 失败）或 commit 数为 0，则：
     - Agent 8（AI 原生度）中"git log 分析"和"最近编写判断"部分标记为 `N/A — 无 git 历史`，仅分析工具链配置和代码结构特征
     - 其他依赖 git 的分析项同理跳过，转为基于文件内容的静态分析
3. **无法识别构建体系**：
   - 如果找不到任何已知的构建/依赖配置文件（如 package.json、go.mod、pom.xml、Cargo.toml、requirements.txt、Makefile 等），则：
     - Agent 9（依赖与工具链）标记为 `N/A — 未检测到标准构建配置`，仅列出发现的文件类型和可能的技术栈
     - Agent 10（新人上手）的"环境搭建"部分如实标注无法确定构建流程，基于文件结构给出最佳猜测
4. **单次尝试原则**：对于任何探测性操作（检测语言、检测框架、读取配置），如果首次尝试失败或返回空结果，**不得循环重试**，直接将该项标记为「未检测到」并继续下一项分析
5. **超时兜底**：如果某个分析子项耗时过长（如遍历超大目录），应在合理范围内采样分析，而非穷举

## 10 个并行分析维度

### Agent 1: 架构分析
- 项目整体目录结构与模块划分
- 核心组件关系与分层架构图（用 ASCII 或 Mermaid）
- 入口文件、配置加载流程、请求处理链路
- 微服务间的调用关系（如适用）

### Agent 2: 调用链与依赖图
- 核心函数/方法的调用链路 map
- 模块间依赖关系（import/require 分析）
- 循环依赖检测
- 外部服务调用点梳理（HTTP、RPC、DB、MQ 等）

### Agent 3: 代码质量评估
- 错误处理模式（是否有裸 panic、吞错误、空 catch 等）
- 测试覆盖情况（有无测试文件、测试风格、覆盖关键路径否）
- 命名规范一致性
- 代码重复度与坏味道（过长函数、God Class、魔法数字等）
- 注释质量与文档字符串

### Agent 4: 风险评估与上手痛点
- 安全风险（硬编码密钥、SQL 注入、不安全的反序列化等）
- 运维风险（缺少健康检查、无优雅停机、日志不足等）
- 新人上手痛点（文档缺失、隐式约定、复杂的本地搭建流程等）
- 技术债务热点

### Agent 5: 优化建议（P0/P1/P2 分级）
基于以上分析，输出分级优化建议：
- **P0（必须立即修复）**：安全漏洞、数据丢失风险、生产事故隐患
- **P1（近期应优化）**：性能瓶颈、错误处理缺陷、测试空白区、架构腐化
- **P2（建议改进）**：代码风格统一、重构建议、开发体验提升
- 每条建议必须包含：问题描述、影响范围、具体修复方案、预估工作量

### Agent 6: 迭代建议与路线图
- 短期迭代建议（1-2 周内可完成）
- 中期改进方向（1-3 个月）
- 长期架构演进建议
- 建议的迭代优先级排序与依赖关系
- 每个迭代项标注预期收益

### Agent 7: 文档建议
- 当前文档现状评估（README、API 文档、架构文档、部署文档）
- 缺失的关键文档清单
- 现有文档的准确性和时效性
- 建议补充的文档及其优先级
- 推荐的文档工具和组织方式

### Agent 8: AI 原生度评估
分析代码库的 AI Native 程度：

> **前置检测**：先运行 `git rev-parse --git-dir` 和 `git log --oneline -1`。如果不是 git 仓库或无 commit 历史，跳过下方"AI 编写痕迹检测"和"最近编写判断"两节，直接标注 `N/A — 无 git 历史`，仅执行"AI Native 成熟度评分"部分。

- **AI 编写痕迹检测**（需 git 历史）：
  - 分析 git log 中的提交模式（提交信息风格、批量修改模式、时间分布）
  - 代码风格一致性分析（AI 生成代码通常过度统一）
  - 注释风格（AI 倾向生成解释性注释）
  - 错误处理模式（AI 倾向面面俱到的 error handling）
  - 变量命名风格（AI 倾向描述性长命名）
- **最近编写判断**（需 git 历史）：分析最近 20 次提交，判断每次提交更可能是：
  - 纯人工编写
  - 人机协作（人写框架 + AI 补充细节，或 AI 生成 + 人审查修改）
  - 主要 AI 生成
- **AI Native 成熟度评分**（1-10，始终执行）：
  - 是否有 AI 辅助开发的工具链配置（.cursor、.claude、copilot 配置等）
  - 是否有 prompt 模板或 AI 协作规范
  - 代码结构是否对 AI 友好（模块化、类型完备、文档充分）
  - 距离 "AI Native" 还差什么，给出具体差距和改进路径

### Agent 9: 依赖与工具链评估

> **前置检测**：搜索常见的依赖/构建配置文件（package.json、go.mod、go.sum、pom.xml、build.gradle、Cargo.toml、requirements.txt、pyproject.toml、Gemfile、Makefile、CMakeLists.txt 等）。如果均不存在，将本维度标记为 `N/A — 未检测到标准构建配置`，仅列出文件扩展名分布和推测的技术栈，不得反复搜索或猜测。

- **依赖清单审计**：列出所有直接依赖及其版本
- **过时依赖检测**：
  - 哪些依赖有更新的主版本可用
  - 哪些依赖已停止维护或被废弃
  - 哪些依赖有已知安全漏洞
- **升级建议**：
  - 每个建议升级的依赖：当前版本 → 建议版本
  - 升级的收益（性能、安全、新特性）
  - 升级的风险（Breaking changes、兼容性问题）
  - 升级优先级排序
- **工具链评估**：
  - 构建工具是否为当前最佳实践
  - CI/CD 配置是否合理
  - 开发工具链是否有更现代的替代方案
  - Linter、Formatter、测试框架是否为社区推荐

### Agent 10: 新人上手指南
为首次接触该项目的开发者提供完整的上手建议：

> **前置检测**：检查是否存在 README、构建配置、Dockerfile 等。如果缺少构建配置，环境搭建部分如实标注「无法从代码自动推断完整的构建流程」，基于文件结构和代码内容给出最佳猜测，不得反复尝试构建或搜索构建方式。

- **环境搭建**：
  - 需要安装哪些工具和依赖（语言版本、包管理器、数据库等）
  - 本地开发环境搭建的完整步骤（从 clone 到跑通第一个请求）
  - 常见的环境搭建坑点和解决方案
- **代码导读路线**：
  - 建议的代码阅读顺序（从哪个文件开始，按什么路径读）
  - 核心概念和领域术语解释
  - 关键设计决策的背景（为什么用这个框架、为什么这样分层）
- **开发工作流**：
  - 日常开发流程（分支策略、提交规范、CR 流程）
  - 如何跑测试、如何本地调试
  - 如何部署和验证改动
- **快速上手任务建议**：
  - 推荐 3-5 个适合新人的 Good First Issue 级别的改进点
  - 每个任务标注预估耗时、涉及文件、学习收益
- **常见问题 FAQ**：
  - 基于代码分析预测新人可能遇到的常见问题
  - 容易踩的坑和隐式约定
